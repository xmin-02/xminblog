## 개요
디버거에서 함수에 멈춰 섰을 때 `RDI`에 무엇이 들어 있는지, 스택 프레임이 어떻게 쌓이는지, `0x7ffe...`로 시작하는 주소가 왜 스택인지를 즉답할 수 있다면 디버깅·리버싱·취약점 분석의 절반은 이미 해결된 상태입니다. 이 글에서는 **x86-64 레지스터 구조**, **System V AMD64 호출 규약**, 그리고 **프로세스 메모리 모델** 세 가지를 한 번에 정리합니다.

가나다라마바사아

## 사전 지식
- C 언어 기본 (포인터, 함수 호출, 배열)
- 16진수와 비트 연산에 대한 감각
- `gcc`, `objdump`, `gdb` 실행 환경 (Linux 기준)
- 스택과 힙의 개념 정도

## 이론
### 핵심 개념

**범용 레지스터 (16개, 64-bit)**

| 그룹 | 레지스터 | System V AMD64에서의 용도 |
| --- | --- | --- |
| 전통적 | `RAX` | 반환값, syscall 번호 |
| | `RBX` | callee-saved (보존) |
| | `RCX` | 4번째 인자 (syscall에서는 RIP 백업용) |
| | `RDX` | 3번째 인자, 보조 반환값 |
| | `RSI`, `RDI` | 2·1번째 인자 |
| | `RBP` | 베이스 포인터, callee-saved |
| | `RSP` | 스택 포인터 |
| 확장 | `R8`, `R9` | 5·6번째 인자 |
| | `R10` | syscall 시 4번째 인자 (RCX 대체) |
| | `R11` | 임시, syscall 시 RFLAGS 백업 |
| | `R12`–`R15` | callee-saved |

64비트 레지스터는 부분 접근이 가능합니다. 예를 들어 `RAX`는 하위 32비트가 `EAX`, 16비트가 `AX`, 8비트가 `AL`(하위) / `AH`(상위 8비트, 단 일부 명령에서만)로 잡힙니다. 중요한 함정: **32비트 레지스터(`EAX` 등)에 쓰면 상위 32비트가 자동으로 0으로 채워집니다.** 이건 8/16비트 부분 쓰기와 다른 동작이라 어셈블리 분석 시 자주 헷갈립니다.

**특수 레지스터**
- `RIP`: 다음에 실행할 명령 주소. `call`/`ret`/`jmp`로만 직접 변경됨
- `RFLAGS`: 비교 결과 플래그(ZF, CF, SF, OF 등)
- `FS`, `GS`: 64비트에서도 살아남은 세그먼트 레지스터. 유저 공간에선 `FS`가 TLS 베이스, 커널에선 `GS`가 per-CPU 데이터 베이스로 쓰임 (`swapgs` 명령으로 유저↔커널 전환)
- `XMM0`–`XMM15`: SIMD/부동소수점 (호출 규약에서 `XMM0`–`XMM7`이 부동소수점 인자 전달에 사용됨)

**System V AMD64 호출 규약 (Linux/macOS/BSD)**

| 항목 | 위치 |
| --- | --- |
| 정수·포인터 인자 1~6 | `RDI`, `RSI`, `RDX`, `RCX`, `R8`, `R9` |
| 정수 반환값 | `RAX` (128비트는 `RDX:RAX`) |
| 부동소수점 인자 1~8 | `XMM0`–`XMM7` |
| 7번째 이후 인자 | 스택 (오른쪽에서 왼쪽으로 push) |
| Caller-saved (호출자 보존) | `RAX`, `RCX`, `RDX`, `RSI`, `RDI`, `R8`–`R11`, `XMM0`–`XMM15` |
| Callee-saved (피호출자 보존) | `RBX`, `RBP`, `R12`–`R15`, `RSP` |
| Red zone | `RSP` 아래 128바이트 (leaf 함수가 RSP 조정 없이 사용 가능) |
| 스택 정렬 | `call` 직전 16바이트 정렬 (`call` 후엔 +8 어긋남) |

> **Linux syscall 규약은 약간 다릅니다.** 인자가 `RDI`, `RSI`, `RDX`, **`R10`**, `R8`, `R9` 순서이고, syscall 번호는 `RAX`에 들어갑니다. 4번째 인자가 `RCX`가 아니라 `R10`인 이유는 `syscall` 명령이 `RCX`에 복귀 주소(`RIP`)를 저장하기 때문입니다. Windows의 Microsoft x64 ABI는 또 다른데, 인자 4개만 레지스터(`RCX`, `RDX`, `R8`, `R9`)로 전달하고 호출자가 32바이트 shadow space를 스택에 미리 잡아둡니다.

**프로세스 가상 메모리 레이아웃 (x86-64 Linux, 4-level paging)**

```
0xFFFF_FFFF_FFFF_FFFF ┐
                      │  커널 공간 (128 TB)
0xFFFF_8000_0000_0000 ┘
                      ▒  Non-canonical hole (사용 불가)
0x0000_8000_0000_0000 ┐
                      │  유저 공간 (128 TB)
                      │   ─ [vsyscall/vDSO]
                      │   ─ 스택 (위 → 아래로 성장)
                      │   ─ mmap 영역 (공유 라이브러리, malloc 큰 청크)
                      │   ─ 힙 (brk, 아래 → 위로 성장)
                      │   ─ .bss      (초기화 안 된 전역)
                      │   ─ .data     (초기화된 전역)
                      │   ─ .rodata   (상수)
                      │   ─ .text     (코드)
0x0000_0000_0040_0000 ┘  (PIE면 0x55... 영역으로 무작위화)
0x0000_0000_0000_0000
```

가상 주소는 사실상 **48비트만** 사용합니다(상위 16비트는 47번 비트의 부호 확장값). 이 규칙을 어긴 주소를 역참조하면 **#GP (general protection fault)** 가 납니다 — 익스플로잇 시도 시 자주 부딪치는 "non-canonical address" 오류가 이것입니다.

### 내부 동작

**함수 호출의 한 사이클**
1. 호출자가 인자를 `RDI`, `RSI`, ... 순으로 채우고, 7번째 이후는 스택에 push
2. `call func` 명령이 다음 명령 주소(복귀 주소)를 스택에 push하고 `RIP`를 `func`로 변경
3. 피호출자가 prologue 실행: `push rbp; mov rbp, rsp; sub rsp, N` — 자기 스택 프레임 확보
4. 본문 실행. callee-saved 레지스터를 쓸 거면 먼저 백업
5. 반환값을 `RAX`에 두고 epilogue: `leave` (= `mov rsp, rbp; pop rbp`) → `ret`
6. `ret`는 스택 top의 복귀 주소를 `RIP`에 pop

**메모리 접근의 강한 모델**

x86-64는 **TSO (Total Store Order)** 라는 비교적 강한 메모리 일관성 모델을 따릅니다. 즉, CPU는 같은 스레드 내 명령 재배치 시 *store→load (다른 주소)* 한 가지 패턴만 허용하고 나머지는 보존합니다. 그래서 ARM/POWER에 비해 lock-free 코드의 메모리 배리어 부담이 적습니다. 단, **NUMA, 캐시 라인 false sharing, 페이지 폴트**는 여전히 성능 함정이 됩니다. 캐시 라인 단위는 보통 64바이트.

## 코드
### 예제 코드 — 호출 규약을 어셈블리로 추적

```c
// calling_conv.c
#include <stdio.h>

long add4(long a, long b, long c, long d) {
    long sum = a + b + c + d;
    return sum;
}

int main(void) {
    long result = add4(10, 20, 30, 40);
    printf("result = %ld\n", result);
    return 0;
}
```

빌드 (PIE 끄고, 디버그 심볼 포함, 최적화 끔):
```bash
gcc -O0 -g -no-pie -fno-asynchronous-unwind-tables \
    -masm=intel calling_conv.c -o calling_conv
objdump -d -M intel --no-show-raw-insn calling_conv | grep -A 25 "<add4>:"
```

### 실행 결과
```
0000000000401126 <add4>:
  401126:  push   rbp
  401127:  mov    rbp,rsp
  40112a:  mov    QWORD PTR [rbp-0x18],rdi    ; a (1번째 인자)
  40112e:  mov    QWORD PTR [rbp-0x20],rsi    ; b (2번째 인자)
  401132:  mov    QWORD PTR [rbp-0x28],rdx    ; c (3번째 인자)
  401136:  mov    QWORD PTR [rbp-0x30],rcx    ; d (4번째 인자)
  40113a:  mov    rdx,QWORD PTR [rbp-0x18]
  40113e:  mov    rax,QWORD PTR [rbp-0x20]
  401142:  add    rdx,rax
  401145:  mov    rax,QWORD PTR [rbp-0x28]
  401149:  add    rdx,rax
  40114c:  mov    rax,QWORD PTR [rbp-0x30]
  401150:  add    rax,rdx                     ; sum 계산 결과
  401153:  mov    QWORD PTR [rbp-0x8],rax     ; 지역변수 sum에 저장
  401157:  mov    rax,QWORD PTR [rbp-0x8]     ; 반환값을 RAX로
  40115b:  pop    rbp
  40115c:  ret
```

읽는 법:
- `push rbp; mov rbp, rsp` → 표준 prologue. 새 프레임의 베이스를 잡음
- `[rbp-0x18]`, `[rbp-0x20]`, ... → `RDI`, `RSI`, `RDX`, `RCX`로 들어온 인자들을 자기 스택 프레임에 백업한 것 (`-O0`이라 일부러 메모리에 저장; `-O2`로 빌드하면 레지스터 안에서 끝나고 함수 자체가 인라인됨)
- 마지막에 `RAX`에 결과 두고 `ret` → 호출 규약 그대로

## 실전 활용
### 활용 예시 — `/proc/self/maps`로 메모리 레이아웃 직접 확인

이론에서 본 그림이 실제 프로세스에 어떻게 그려지는지 확인합니다.

```c
// memory_map.c
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>

int  global_init = 0xAA;     // .data
int  global_bss;             // .bss
const char *rodata_str = "rodata!";   // 포인터는 .data, 문자열은 .rodata

int main(void) {
    int   stack_var = 0x55;          // 스택
    int  *heap_var  = malloc(sizeof *heap_var);  // 힙
    *heap_var = 0x77;

    printf("text  (main)        : %p\n", (void *)main);
    printf("rodata (string)     : %p\n", (void *)rodata_str);
    printf("data  (global_init) : %p\n", (void *)&global_init);
    printf("bss   (global_bss)  : %p\n", (void *)&global_bss);
    printf("heap  (malloc)      : %p\n", (void *)heap_var);
    printf("stack (local)       : %p\n", (void *)&stack_var);
    printf("libc  (printf)      : %p\n", (void *)printf);
    printf("\n--- /proc/%d/maps ---\n", getpid());
    fflush(stdout);

    char cmd[64];
    snprintf(cmd, sizeof cmd, "cat /proc/%d/maps", getpid());
    system(cmd);

    free(heap_var);
    return 0;
}
```

```bash
gcc -O0 -no-pie memory_map.c -o memory_map
./memory_map
```

### 실행 결과

체크 포인트:
- **`r-xp`** 권한이 붙은 영역이 코드 영역 — 여기에 쓰기가 가능했다면 코드 인젝션이 사소한 문제가 됐겠죠. W^X 원칙(쓰기와 실행은 동시에 안 됨)이 페이지 권한으로 강제됩니다.
- 힙은 `0x1d4...`, 스택은 `0x7ffd...`, libc는 `0x7f8c...` — 주소대를 보면 어디 영역인지 즉시 분간이 됩니다. PIE를 켜면 코드 영역도 `0x55...` 대역으로 무작위화됩니다.
- `[vsyscall]`은 정해진 고정 주소 `0xffffffffff600000`에 있는 레거시 매핑. 한때 ROP 공격의 단골 가젯 소스였고, 그래서 최근 커널은 emulate 모드로 두거나 비활성화합니다.

`gdb`로 디버깅 중이라면 `info proc mappings`로 같은 정보를 더 깔끔하게 볼 수 있고, 임의 주소의 소속 영역은 `info symbol 0x7ffd3e2b9b94`로 추적할 수 있습니다.

## 정리
> - 인자는 `RDI, RSI, RDX, RCX, R8, R9` 순서, 반환값은 `RAX`. callee-saved는 `RBX, RBP, R12–R15`. (Linux syscall은 4번째가 `R10`)
> - 가상 주소는 48비트만 유효하고, 0x0000... 대역은 유저 / 0xFFFF8... 대역은 커널, 그 사이는 사용 불가(non-canonical) 영역.
> - `objdump -d -M intel`과 `/proc/self/maps`만 자유롭게 읽을 수 있어도 디버깅·리버싱의 70%는 해결됩니다.
